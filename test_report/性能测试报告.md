# OpenClaw WebSocket 性能测试报告

测试对象：OpenClaw 双节点 WebSocket 压力与稳定性场景
数据来源：`automation_assets/client.js`、`automation_assets/server.js`、`test_case_base/performance_case` 本地测试用例与配置；2GB 内存可用性、24 小时稳定性、PERF-003 心跳性能测试、内存泄漏检测、PERF-005 OOM 风险评估、PERF-006 多维并发曲线和 PERF-007 版本回滚性能基线结果来自用户粘贴的 OpenClaw 微信测试输出、CSV 数据及附件日志。

## 第 1 章 测试结论

### 1.1 结论摘要

本轮 OpenClaw 性能测试方案采用两台 OpenClaw 节点通过 WebSocket 建立连接：一台节点运行 `automation_assets/server.js` 作为压力测试服务端，另一台节点运行 `automation_assets/client.js` 作为压力测试客户端。客户端按配置文件持续发送 `ping`、`echo` 或 `load_test` 类型消息，服务端返回响应并输出连接数、消息数、进程内存、系统内存和 CPU load 等监控信息。

## 第 2 章 测试环境

### 2.1 测试拓扑

| 角色       | 节点            | 脚本                            | 主要职责                                                                  |
| ---------- | --------------- | ------------------------------- | ------------------------------------------------------------------------- |
| 压测客户端 | OpenClaw 节点 A | `automation_assets/client.js` | 建立 WebSocket 连接，按间隔发送测试消息，统计成功数、失败数、延迟分位数。 |
| 压测服务端 | OpenClaw 节点 B | `automation_assets/server.js` | 监听 WebSocket 请求，返回响应，输出连接数、消息数、内存与 CPU 指标。      |

### 2.2 脚本能力说明

`client.js` 支持命令行参数和 JSON 配置文件。核心配置包括：

| 参数               | 含义                        | 默认或示例                           |
| ------------------ | --------------------------- | ------------------------------------ |
| `target`         | WebSocket 服务端地址        | `ws://10.0.12.4:8080`              |
| `duration`       | 压测时长，单位秒            | `60`、`300`、`3600`、`86400` |
| `interval`       | 消息发送间隔，单位毫秒      | `50`、`100`、`500`             |
| `messageType`    | 消息类型                    | `ping`、`echo`、`load_test`    |
| `waitForWelcome` | 是否等待服务端 welcome 消息 | `true`                             |
| `welcomeTimeout` | welcome 等待超时            | `5000ms`                           |

客户端输出指标包括成功请求数、失败请求数、总请求数、成功率、平均延迟、最小延迟、最大延迟、P50、P95、P99 和错误数量。

`server.js` 支持 `ping`、`echo`、`load_test`、`get_metrics` 和 `shutdown` 消息类型，并定期输出：

| 指标组   | 指标                           |
| -------- | ------------------------------ |
| 连接指标 | 当前活跃连接数、累计连接数     |
| 消息指标 | 累计接收消息数、累计发送消息数 |
| 进程内存 | RSS、heapUsed、heapTotal       |
| 系统内存 | total、free、used              |
| CPU      | `os.loadavg()`               |

### 2.3 公平性与一致性要求

为保证结果可比，所有场景应保持以下条件一致：

| 维度     | 要求                                                                   |
| -------- | ---------------------------------------------------------------------- |
| 网络     | 两台 OpenClaw 节点使用稳定内网连接，避免跨公网或代理链路引入额外波动。 |
| 脚本版本 | client/server 脚本版本固定，测试过程中不修改统计逻辑。                 |
| 目标地址 | 同一轮测试使用相同 `target`，默认 `ws://10.0.12.4:8080`。          |
| 运行环境 | 记录 CPU、内存、操作系统、容器内存限制和 Node.js 版本。                |
| 采集口径 | 每个用例保存客户端最终报告、服务端监控日志和系统级监控结果。           |

## 第 4 章 执行方法

### 4.1 服务端启动

在服务端 OpenClaw 节点执行：

```bash
node automation_assets/server.js --port=8080 --host=0.0.0.0 --log-level=info
```

服务端启动后应确认监听地址为 `ws://0.0.0.0:8080`，并观察是否周期性输出系统监控指标。

### 4.2 客户端执行

在客户端 OpenClaw 节点执行基础压测：

```bash
node automation_assets/client.js --config test_case_base/performance_case/test-config.json
```

执行专项用例时替换 `--config` 指向对应配置文件，例如：

```bash
node automation_assets/client.js --config test_case_base/performance_case/heartbeat-spike-config.json
node automation_assets/client.js --config test_case_base/performance_case/memory-leak-config.json
node automation_assets/client.js --config test_case_base/performance_case/24h-stability-config.json
```

### 4.3 OpenClaw 微信侧通过 GitHub API 读取执行

除在测试节点本地手工执行脚本外，本轮测试也支持通过 OpenClaw 微信侧接入 GitHub API，由 OpenClaw 直接读取仓库中的压测脚本和测试用例配置，再在目标节点完成测试调度与结果回传。该方式适用于不方便手工拷贝脚本、测试用例频繁迭代、需要在微信侧统一发起和记录测试结果的场景。


微信侧执行流程：

## 第 6 章 分析与发现

### 6.1 已确认事实

L1 确定性结论，数据来源：本地脚本和配置文件。

OpenClaw 当前性能测试链路已经具备基础自动化能力。`client.js` 能够从本地 JSON 配置读取目标地址、测试时长、发送间隔、消息类型和 welcome 等待策略，并在测试结束时输出成功率和延迟分位数。`server.js` 能够响应 `ping`、`echo`、`load_test` 请求，并输出基础资源监控。

L1 确定性结论，数据来源：`test_case_base/performance_case`。

测试用例覆盖了发布前性能验证中最重要的资源风险：低内存可用性、长时间稳定性、心跳突刺、内存泄漏、OOM 风险、并发容量曲线和版本变更性能回退。

L1 确定性结论，数据来源：用户粘贴的 2GB 内存可用性测试结果。

2GB 内存可用性测试共处理 598 个请求，成功 598 个、失败 0 个，成功率 100.00%；平均延迟 1.82ms，P95 延迟 2ms，P99 延迟 3ms，最大延迟 6ms。server.js 占用 51.09MB，堆内存使用 5.59MB，在 1936MB 系统总内存环境下资源占用较低。

L1 确定性结论，数据来源：用户提供的 24 小时稳定性附件日志。

24 小时稳定性测试使用 `24h-stability-config.json`，目标地址为 `ws://10.0.12.4:8080`，持续 86,400 秒，消息间隔 500ms，消息类型为 `ping`。本次测试共处理 172,555 个请求，成功 172,555 个、失败 0 个，成功率 100.00%；平均延迟 1.54ms，最小 1ms，最大 10ms，P50 2ms，P95 2ms，P99 2ms。附件中的服务端启动阶段资源基线为 RSS 53.11MB、heapUsed 5.77MB、heapTotal 7.10MB、CPU load 0.04/0.02/0。

L1 确定性结论，数据来源：用户粘贴的 PERF-003 心跳性能测试结果。

PERF-003 心跳性能测试共处理 174,689 个请求，成功率 100.00%，平均延迟 1.45ms，QPS 582.20；Gateway CPU 为 0.8%，未检测到日志风暴、Retry Storm 或 Gateway 不可用。

L1 确定性结论，数据来源：用户粘贴的内存泄漏检测测试结果。

内存泄漏检测运行 30 分钟，采集 60 个监控样本，WebSocket 连接 177,738 次。初始 RSS 为 385,516KB，峰值 RSS 为 406,804KB，峰值增长 5.52%；最终 RSS 为 384,908KB，较初始值下降 0.15%。测试未检测到持续增长、无法回收或泄漏迹象，结论为无内存泄漏。

L1 确定性结论，数据来源：用户粘贴的 PERF-005 OOM 风险评估测试结果。

PERF-005 OOM 风险评估运行 18 分钟，覆盖 50%、60%、70%、80%、85% 五个压力阶段，共 90 个采样点。测试期间 OOM 事件为 0 次，90 个采样点状态均为 RUNNING；RSS 范围为 417,720-422,708KB，即 407-412MB，系统空闲内存为 1077-1096MB。由于 OOM Score 全程为 758，且内存配置约 870MB 偏紧，发布评估结论为有条件发布。

### 7.1 测试执行建议
