### 3.3 Issue #96389 多页扫描 PDF 图片 fallback 复现

问题：`document-extract` 在文本提取失败后 fallback 到 `mode: "images"` 时，`clawpdf` 的 4M 像素预算在多页间共享，导致多页扫描 PDF 后续页面被渲染成极小图片或不被处理。

复现环境：

```text
OpenClaw 2026.5.7
clawpdf 0.3.0
Node.js v22.22.2
```

复现结果：

| 项 | 结果 |
|---|---|
| 测试 PDF | 15 页, 2480x3508 点 |
| extract 模式 | `mode: "images", image.maxPixels: 4000000` |
| 输出图片数 | 2 |
| Page 1 | 1682x2378, 3,999,796 像素, 正常 |
| Page 2 | 12x17, 204 像素, 异常 |
| Page 3-15 | 未处理 |

结论：Bug 已确认复现。建议将 `maxPixels` 改为每页独立计算，或在 OpenClaw 侧改为逐页渲染 workaround。

### 3.4 Issue #96704 托管浏览器 Cookie 持久化复现

问题：OpenClaw 托管浏览器通过 CDP 控制产生的 cookie 只保存在内存中，不持久化到磁盘 SQLite 数据库，导致浏览器或 gateway 重启后登录态丢失。

复现环境：

```text
OpenClaw 2026.6.10
Chrome 147.0.7727.15
Linux 6.8.0-101-generic x64
Node v22.22.2
```

复现结果：

| 指标 | 访问前 | 访问后 | 变化 |
|---|---:|---:|---|
| Cookie 文件大小 | 20480 bytes | 20480 bytes | 无变化 |
| Cookie 修改时间 | 2026-06-19 13:34:56 | 2026-06-19 13:34:56 | 无变化 |

结论：Bug 已确认复现。该问题会影响依赖浏览器登录态、SSO/MFA 的自动化任务，建议实现 cookie 保存/恢复机制和定期刷新机制。

#### 3.5.2 主要优点

1. 能快速生成常见代码模板。
2. 对基础 Python、Go、Node.js/Express、FastAPI 有一定熟悉度。
3. 能写出基本类型提示、docstring、注释和项目结构。
4. 对典型问题有基本认识，例如列表去重优化、全负数最大值处理、统计请求不应阻塞重定向、WebSocket 水平扩展可借助消息总线。
5. 能产出 Docker Compose、k6、README 等工程外围材料。

#### 3.5.3 核心缺点

| 问题 | 风险 |
|---|---|
| 验证不可信 | 多处声称测试通过, 但缺少真实运行日志、命令输出和失败处理 |
| README 与代码实现不一致 | WebSocket 项目声称 ACK、重连补发、at-least-once, 但代码未闭环 |
| 分布式理解偏表层 | 短链接服务用 Redis `INCR` 中心化计数器, 未满足分布式唯一性约束 |
| 并发安全不足 | WebSocket 多 goroutine 写连接、Hub map 并发访问存在风险 |
| 高并发指标缺少证明 | 10,000 TPS、P99、at-least-once 等指标无可信压测支撑 |
| 可靠性机制不闭环 | 统计 flush、Redis Pub/Sub、Stream ACK/补发等关键机制缺失 |
| 测试设计偏浅 | 缺少异常路径、并发测试、故障注入、重启恢复和一致性测试 |

#### 3.5.4 高难度工程任务结论

短链接服务具备 demo 骨架，但未满足生产级高并发和分布式可靠性要求：

- Redis `INCR` 不符合“不依赖单机自增 ID”的要求。
- 写入链路难以证明 10k TPS。
- 统计异步 flush 存在丢数据风险。
- 缓存热点策略和异常恢复不足。
- k6 脚本不能证明目标达成。

WebSocket 实时推送系统具备基础推送 demo，但不满足高可靠和高并发要求：

- at-least-once 未真正实现。
- WebSocket 并发写风险严重。
- Hub map 并发访问风险明显。
- 客户端无自动重连。
- Redis Pub/Sub 无持久化, Stream 未闭环。
- 延迟压测指标不可信。

#### 3.5.5 适用与不适用场景

适用：

- 简单脚本。
- 常见函数实现。
- 代码片段生成。
- demo 原型。
- 基础测试样例。
- 初版项目骨架。

不适用：

- 直接交付生产级服务。
- 高并发系统核心实现。
- 分布式可靠性机制。
- 需要真实压测证明的任务。
- 金融、支付、消息、计费、审计等高可靠场景。

结论：OpenClaw 的编码能力在低中难度任务中可用，但在高难度工程任务中主要停留在“搭骨架”和“写术语”的层面。它尚不具备独立完成高并发、分布式、可靠消息系统核心代码的能力。

结论：在 2 核 2 GB 环境下，OpenClaw 技能系统文件层表现优秀，支持 200+ 并发无崩溃。但该测试仅覆盖文件读取层面，不能替代真实浏览器、API、视频处理等完整技能调用压测。


## 4. 风险与问题

### 4.1 已确认产品问题

1. Issue #96389: 多页扫描 PDF fallback 图片渲染异常。
   - 影响文档抽取和视觉模型下游调用。
   - 多页高分辨率扫描件风险高。

   - 影响登录态保持、SSO/MFA、无人值守浏览器自动化任务。

### 4.2 代码能力风险

1. 高难度任务中多项硬性要求未满足, 代码无法真实满足生产级需求。
2. WebSocket 写入和 map 访问存在并发安全风险。

4. TPS、P99、at-least-once 等指标缺少可信验证, 存在指标虚标风险。

### 5.测试中发现的问题

###  mock-blackhole 超时测试后的微信会话恢复问题

测试目的：验证真实模型/provider idle timeout 是否能被正确识别为 `timedOut=true`，而不是工具或运行时 abort 场景中的 `aborted=true, timedOut=false`。

触发方式：将默认模型临时配置为永远不返回首 token 的黑洞模型：

```text
mock-blackhole/blackhole-model
```

问题现象：

- Weixin 对话持续返回 `Something went wrong while processing your request`。
- `/new` 可以创建新会话，但普通消息仍无法恢复。
- 恢复 `agents.defaults.model.primary` 并重启 gateway 后，仍出现模型路由残留或错误模型解析。
- 排查过程中曾出现 `Unknown model: volcengine-plan/deepseek-v4-flash`、`Unknown model: tencent-plan/tc-code-latest` 等日志。

最终恢复方式：确认真实 provider/model 为：

```text
tencenttokenplan/tc-code-latest
```

并在 Weixin 会话中显式执行：

```text
/model tencenttokenplan/tc-code-latest
```

系统返回：

```text
Model reset to default (tencenttokenplan/tc-code-latest).
```

结论：该问题不是单纯的模型超时，而是黑洞模型测试后暴露出的 session/model route 状态恢复问题。GitHub 机器人反馈显示，current main 已包含 stale session/model route 清理逻辑，但该修复尚未进入当前 release tag；在当前版本中仍需要手动通过 `/model <provider>/<model>` 恢复会话模型状态。

风险影响：

- timeout 对照测试可能污染后续 Weixin 会话。
- `/new` 和 gateway restart 不一定能清理显式模型 pin 或旧 session model route。
- 通用错误文案缺少可操作信息，容易误判为普通会话故障。
