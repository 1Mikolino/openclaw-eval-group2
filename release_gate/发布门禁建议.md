# 发布门禁建议

> **组职责**：业务场景测试 + 性能稳定性测试
>
> **门禁目标**：确保 OpenClaw 在业务场景和性能方面的质量

---

## 一、门禁概述

### 1.1 门禁定位

第 2 组的质量门禁专注于**两个核心维度**：

```
┌─────────────────────────────────────────┐
│         第2组质量门禁体系              │
├─────────────────────────────────────────┤
│                                         │
│  🎯 业务场景门禁                        │
│  ├─ 编码开发能力测试                    │
│  ├─ 多浏览器并发测试                   │
│  ├─ Top10 Skill 崩溃测试              │
│  └─ Web工具指南测试                    │
│                                         │
│  ⚡ 性能稳定性门禁                      │
│  ├─ 24小时稳定性测试                   │
│  ├─ 内存泄漏检测                       │
│  ├─ OOM风险评估                        │
│  ├─ 心跳峰值测试                       │
│  ├─ 多维度并发测试                    │
│  └─ 版本回滚性能测试                  │
│                                         │
└─────────────────────────────────────────┘
```

### 1.2 与全局门禁的关系

```
全局门禁（其他组负责）
├─ 基础CI门禁
├─ 安装冒烟测试
├─ 跨平台验证
└─ ...

第2组门禁（本方案）
├─ 业务场景门禁 ✅
└─ 性能稳定性门禁 ✅
```

**协作机制**：

- 全局门禁通过后，进入第 2 组专项门禁
- 第 2 组门禁通过，才能发布到 `stable` 版本
- 第 2 组门禁失败，阻塞发布并通知第 2 组

## 二、门禁维度详细设计

### 2.1 业务场景门禁（Business Scenario Gates）

#### **门禁 BS-001：编码开发能力测试**

**对应测试用例**：`business_scenarios/coding_test_case/`

**测试内容**：

- A 类：基础函数实现（字母异位词分组、时长格式化等）
- B 类：数据结构与算法（LRU 缓存、TopK 问题等）
- C 类：实际开发任务（CLI 工具、API 服务等）

**门禁判定标准**：

| 测试项       | 权重 | 通过标准                         |
| ------------ | ---- | -------------------------------- |
| 需求理解     | 15%  | 准确理解需求，无明显偏差         |
| 代码正确性   | 20%  | 通过所有基础测试用例             |
| 边界条件     | 15%  | 正确处理空输入、异常值、边界情况 |
| 工程结构     | 15%  | 代码结构清晰，符合工程规范       |
| 测试验证     | 15%  | 提供有效测试代码和测试输出       |
| 性能与可靠性 | 10%  | 无明显性能问题，无崩溃风险       |
| 可维护性     | 10%  | 有清晰的说明和注释               |

**通过条件**：

- ✅ 综合评分 ≥ 70 分
- ✅ 代码正确性必须 100%通过
- ✅ 无 P0 级别缺陷

**执行方式**：

```yaml
- name: 编码开发能力测试
  run: |
    # 1. 运行编码测试用例库
    python run_coding_tests.py --test-dir coding_test_case

    # 2. 评分
    python evaluate_coding_results.py --output results.json

    # 3. 判定
    if [ $(jq '.overall_score' results.json) -ge 70 ]; then
      echo "✅ 编码能力测试通过"
    else
      echo "❌ 编码能力测试失败"
      exit 1
    fi
```

### 2.2 性能稳定性门禁（Performance & Stability Gates）

#### **门禁 PS-001：24 小时稳定性测试**

**对应测试用例**：`performance_case/openclaw_24h_stability_test.md`

**测试内容**：

- 连续运行 24 小时
- 监控内存、CPU、磁盘使用情况
- 验证长时间运行的稳定性

**门禁判定标准**：

```yaml
stability_criteria:
  memory_growth:
    max_growth_rate: 5% # 每小时内存增长不超过5%
    leak_detected: false # 未检测到内存泄漏

  cpu_usage:
    avg_usage: < 30% # 平均CPU使用率 < 30%
    peak_usage: < 80% # 峰值CPU使用率 < 80%

  error_rate:
    max_error_rate: 0.1% # 错误率 < 0.1%
    crash_count: 0 # 崩溃次数 = 0

  connection_stability:
    ws_disconnect_rate: < 1% # WebSocket断开率 < 1%
    reconnect_success: ≥ 99% # 重连成功率 ≥ 99%
```

**通过条件**：

- ✅ 所有指标在阈值范围内
- ✅ 无 P0/P1 级别问题
- ✅ 日志文件大小合理（< 10GB）

**执行方式**：

```yaml
- name: 24小时稳定性测试
  timeout-minutes: 1440 # 24小时
  run: |
    # 1. 启动监控
    ./scripts/start_monitoring.sh

    # 2. 运行24小时测试
    python run_24h_stability_test.py \
      --config performance_case/24h-stability-config.json

    # 3. 生成报告
    python generate_stability_report.py \
      --output stability_report.json

    # 4. 判定
    python check_stability_criteria.py \
      --report stability_report.json \
      --criteria stability_criteria.json
```

### 3.2 门禁分组执行

参考 OpenClaw 的 `rerun_group` 设计，支持灵活执行：

```yaml
inputs:
  gate_group:
    description: "选择要运行的门禁组"
    required: false
    default: "all"
    type: choice
    options:
      - all # 运行所有门禁
      - business # 仅业务场景门禁
      - performance # 仅性能稳定性门禁
      - bs-001 # 仅编码能力测试
      - ps-001 # 仅24小时稳定性测试
```

**使用场景**：

- **日常验证**：运行 `business` 组，快速验证业务场景
- **发布前**：运行 `all`，全量门禁
- **问题定位**：运行特定门禁（如 `ps-002`），针对性验证

## 五、GitHub Actions 工作流示例

### 5.1 主工作流文件：`.github/workflows/group2-quality-gates.yml`

```yaml
name: Group2 Quality Gates

on:
  workflow_dispatch:
    inputs:
      gate_group:
        description: "门禁组选择"
        required: false
        default: "all"
        type: choice
        options:
          - all
          - business
          - performance
          - bs-001
          - bs-002
          - bs-003
          - bs-004
          - ps-001
          - ps-002
          - ps-003
          - ps-004
          - ps-005
          - ps-006
      ref:
        description: "待测版本分支/SHA"
        required: true
        type: string

  schedule:
    # 每天凌晨2点运行全量门禁
    - cron: "0 2 * * *"

env:
  TEST_CASE_REPO: "https://github.com/1Mikolino/openclaw-eval-group2.git"
  TEST_CASE_BRANCH: "main"

jobs:
  # 阶段1：业务场景门禁
  business_scenario_gates:
    runs-on: ubuntu-24.04
    timeout-minutes: 300
    if: contains(fromJSON('["all","business","bs-001","bs-002","bs-003","bs-004","bs-005"]'), inputs.gate_group)
    strategy:
      matrix:
        gate:
          - id: bs-001
            name: 编码开发能力测试
            path: business_scenarios/coding_test_case
```
